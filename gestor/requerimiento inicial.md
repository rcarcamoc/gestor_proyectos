SmartTrack — Documento Técnico de Requerimiento Versión 2.0 — Definitiva para Inicio de Desarrollo Estado: Aprobado para desarrollo Fecha: Abril 2026 Alcance: MVP completo, ejecución por fases Primera entrega: Docker local + estructura base

Resumen ejecutivo y criterios de diseño técnico SmartTrack es una plataforma web de gestión de proyectos y carga de trabajo para equipos, cuyo diferenciador principal es un motor de scheduling inteligente que opera incluso con datos incompletos, anticipando conflictos y recomendando acciones correctivas de forma operativa.
Tres reglas que rigen todas las decisiones del equipo Regla 1 — Utilidad con datos incompletos El sistema debe ser útil incluso cuando faltan horarios, skills o historial. Degradar con advertencias siempre es preferible a bloquear.

Regla 2 — Transparencia del motor La UX debe mostrar claramente cuándo una recomendación es estimada vs. precisa. El nivel de confianza del motor debe ser visible en todo momento.

Regla 3 — Validar antes de optimizar No se debe optimizar infraestructura cloud antes de validar el flujo real de uso. El MVP se construye y valida localmente primero.

Propósito del producto SmartTrack permite a líderes y miembros de equipo gestionar proyectos, asignaciones y carga de trabajo con visibilidad real del impacto operacional. El motor no solo registra tareas: evalúa disponibilidad, prioridades, skills y fechas para anticipar problemas antes de que ocurran.
El enfoque del MVP prioriza:

Adopción rápida (equipo operativo < 10 minutos). Utilidad desde el primer uso, incluso con datos incompletos. Experiencia de producto, no de base de datos. Valor visible desde el onboarding. 2. Autenticación — JWT propio Decisión confirmada: JWT con implementación propia en backend. No se usará Firebase Auth ni Auth0 en el MVP.

Motor inteligente — Principios no negociables El motor nunca bloquea. Siempre degrada con advertencias y nivel de confianza explícito.

10.1 Capacidades del motor (MVP) Capacidad Descripción Detección de sobrecarga Evalúa si un miembro tiene horas insuficientes Evaluación de deadline Estima si el proyecto queda comprometido Reasignación sugerida Propone otro miembro según skills y/o disponibilidad Reprogramación sugerida Propone mover fecha de inicio u objetivo Impacto colateral Informa qué tareas se atrasan si se fuerza la asignación Estimación automática Infiere valores cuando faltan horas o fechas 10.2 Niveles del modo degradado Nivel Datos presentes Comportamiento FULL Horario + skills + historial Máxima precisión, recomendación completa HIGH Horario + algunos skills Alta precisión, con limitaciones menores BASIC Solo miembros y tareas mínimas Usa 8h/día por defecto, disponibilidad estimada DEGRADED Datos mínimos o casi nulos Usa 6h/día conservadoras, heurísticas simples 10.3 Defaults del motor — Configuración, no constante Decisión confirmada: Los valores 8h/día (BASIC) y 6h/día (DEGRADED) no se hardcodean como constantes de código. Se persisten como configuración editable.

MVP: Los defaults se guardan como configuración global de sistema en base de datos (tabla system_config). No son constantes en código.

Fase futura: El Owner de organización podrá editar estos valores desde la UI sin cambios de código.

10.4 Reglas operativas del motor degradado Si faltan horas estimadas: inferir por historial de tareas similares; si no hay historial, usar 4h por defecto. Si falta fecha de inicio: usar la próxima fecha laborable disponible. Si no hay horario configurado: usar el default según nivel (BASIC u DEGRADED). Si no hay skills: sugerir por disponibilidad o menor carga activa, indicando que la recomendación es aproximada. 10.5 Respuesta estándar del motor Toda respuesta del motor debe incluir esta estructura, independientemente del nivel de datos:

Modo emergencia Requisitos mínimos Analizar impacto global de una ausencia, bloqueo o cambio urgente Mostrar preview antes de aplicar cambios Generar snapshot completo del estado anterior Permitir undo de la reprogramación en un clic dentro de una ventana de 2 horas (default) Registrar: quién ejecutó, cuándo, por qué y qué entidades fueron afectadas Ventana de rollback Decisión confirmada: Default de 2 horas para la ventana de rollback en el MVP.

Diseño: La ventana se almacena como configuración en base de datos (no hardcodeada). En fases futuras, el Owner de organización podrá modificarla.
